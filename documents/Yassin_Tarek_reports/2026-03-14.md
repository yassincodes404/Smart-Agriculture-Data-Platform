# Smart Agriculture Data Platform - Engineering Report
**Date:** March 14, 2026

**Subject:** Full Architectural Overhaul, Microservices Containerization, and Component Integration

## 🏗️ 1. Project Initialization & Modular Restructuring
We initiated a complete redesign of the Smart Agriculture Data Platform to transition it into a modern, scalable microservices architecture. 
- **Root Hierarchy Reborn:** The project was cleanly separated into `services/` (application logic), `infra/` (gateway and deployment logic), and `data/` (raw datasets and persistent volumes).
- **Stateless Refactoring:** Intentionally stripped out unnecessary background job processors (Celery, Workers, Queue) and message brokers (Redis) to prioritize a lightweight, stateless REST API handling data requests directly. Removed Object storage / MinIO components to focus on relational database integration.
- **Archive Migration:** Legacy files and flat structures from the previous root were securely archived to prevent clutter.

## 🐳 2. Multi-Container Docker Orchestration
Designed a comprehensive `docker-compose.yml` that networks five isolated components together seamlessly:
- **`agri_nginx`**: Nginx Gateway routing `:80` traffic out to the services.
- **`agri_frontend`**: React + Vite client exposed securely.
- **`agri_backend`**: Python backend mapped to internal volume endpoints.
- **`agri_mysql`**: Persistent Relational database avoiding host conflicts via port `3307` mapped to `3306`.
- **`agri_cv`**: A standalone processing environment configured with `tty: true` to prevent container termination while executing OpenCV and Numpy image processing scripts.

## 🐍 3. FastAPI Backend Foundation
Wrote the core foundational layers for the Python Backend:
- **Application Entrypoint**: Wrote `main.py` configuring the core FastAPI `app` object and linked internal router files (e.g., resolving Uvicorn missing `app` assignment and `ImportError`).
- **Health Checks**: Implemented standard `/api/health` and `/api/health/db` checkpoints allowing orchestration environments to monitor container and database connection statuses.
- **Docker Building Reliability**: Boosted pip's `--default-timeout=1000` to prevent timeouts when downloading heavy data engineering dependencies (Pandas, SciPy, Scikit-learn, OpenCV, etc.) into the container.
- **Automated Testing Suite**: Initialized Pytest unit tests mocking database connections and health endpoints (`Test_Back_End_heath.py`, `test_database.py`, `test_items_api.py`).

## 🗄️ 4. Dedicated Database Architecture & Authentication 
We abstracted the raw MySQL container image into a dedicated `Database/` build directory to allow for advanced lifecycle management:
- **Automated Schema Initialization**: Linked an `init.sql` script into the container's `/docker-entrypoint-initdb.d/` folder so MySQL automatically builds out the initial `agriculture` schema and `test_connection` tables upon fresh boot.
- **Connection Logic Resiliency**: Configured SQLAlchemy backend sessions to dynamically retrieve `DATABASE_URL` strings from the environment (`.env.backend`), falling back exactly to the internal container DNS (`mysql`).
- **MySQL 8 Caching SHA-2 Identity Fix**: Addressed a severe authentication hurdle encountered between modern MySQL 8 instances and Python `pymysql` connectors by installing the `cryptography` library, granting native execution over `caching_sha2_password` systems.

## ⚛️ 5. React + Vite Frontend Migration
Moved the web client rendering stack off of heavy Webpack implementations into Vite for significant performance gains:
- **Vite Setup**: Executed `create-vite` within the `services/frontend/web` directory using the React template.
- **Node Modules Mapping**: Fixed a critical Docker issue where local empty directories override container modules by explicitly mounting an anonymous `/app/node_modules` volume in `docker-compose.yml`.
- **App Connectivity**: Sculpted `App.jsx` to natively invoke asynchronous `fetch` requests towards the backend.

## 🌐 6. Environment Connectivity & Hot Module Rebuilding
Addressed multiple developer-experience bugs common in WSL2/VirtualBox/Windows host environments:
- **Vite Hot Module Replacement (HMR)**: Because native file-save hooks don't pass into WSL2 containers from Windows, we mandated `CHOKIDAR_USEPOLLING=true` and `WATCHPACK_POLLING=true` directly within `docker-compose.yml` and overrode `vite.config.js` (`watch.usePolling: true`) to physically poll files every 1000ms. Code edits now instantly appear in the browser.
- **Nginx API Trimming Bug**: Modified the Nginx reverse proxy `default.conf` to strip the trailing slash off the `proxy_pass` directive (`http://backend:8000`), resolving recurring `404 Not Found` errors caused by Nginx improperly rewriting the `/api/` path string before handing it to FastAPI. 
- **WebSocket Tunneling**: Engineered strict WebSocket HTTP Header connection overrides inside Nginx (`Upgrade $http_upgrade`, `Connection "upgrade"`), allowing Vite's hot-reload listener socket to permeate across the proxy network boundary properly to the end user.
